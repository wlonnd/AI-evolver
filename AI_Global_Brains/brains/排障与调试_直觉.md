# 排障与调试直觉 (Brains)

> 本文件汇集了在排查死锁、定位雪崩、清理脏数据以及干预系统运行状态时的高维思维。

## 1. 遗留错误标记陷阱
- **直觉**：在异步队列或批处理系统中，发现界面或数据库存在“历史报错/Error”标记时，**不要立刻主观假设**当前系统仍处于卡死状态或队列正在严重阻塞。
- **应对**：必须精确区分“已被标记为 error 且不再重试的死信遗留态”和“当前正在 pending 的真实堆积态”。遇到此类情况，先编写只读的诊断脚本（如对账 `pending` 数量和最新日志），查清系统的真实存活状态，不要盲目去执行清理操作或强行把死信重入队。

## 2. 自动选路与手动干预的竞态
- **直觉**：当底层系统具备自动决策能力（如网络多路代理的低延迟测速选路、动态负载均衡）时，外部管理脚本（如自愈 Bot）不要轻易使用“绝对固定（Pin/Fix）”逻辑来主动抢占控制权。这会导致系统锁死，无法在网络恢复或拓扑变化时自动回退到最优决策。
- **应对**：正确的自愈思维不是去“固定它”，而是去“释放它”。清理残留的死锁状态，解除旧的固定约束（如 `DELETE /proxies/xxx`），然后让系统重新触发内置的决策流（重测 `delay`），从而恢复选路弹性。

## 3. 多层系统的数据边界隔离漏扫
- **直觉**：在上下游构成的多层过滤系统中（例如：轻量级 Web 上游 -> 重度 GPU 特征扫描引擎下游），如果业务上新增了一类“需要跳过/免除扫描”的特殊数据源（如长视频）。
- **应对**：不能仅仅在前端页面或入口层增加过滤条件。**必须地毯式审查**所有涉及底层 SQL 或消费组件的地方，确保“免除规则”不仅在展示层生效，更在底层高耗能队列被硬过滤（如 `WHERE filepath NOT LIKE '%/long/%'`）。哪怕漏掉一层，都极易引发重计算资源的严重泄露或卡死，且这种问题往往在轻量监控页上是隐形的。
