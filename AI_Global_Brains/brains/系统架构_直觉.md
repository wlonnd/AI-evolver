# 系统架构直觉 (Brains)

> 本文件汇集了跨语言、跨技术栈的高维度架构设计原则与系统防御直觉。

## 1. 持久化与缓存一致性 (长生命周期守护进程)
- **直觉**：在内存中有缓存的进程（如长时间运行的 Bot 守护进程），直接从外部通过脚本修改其依赖的磁盘状态文件是**无效且危险**的。因为守护进程在下一次落盘保存时，会用其内存里陈旧的缓存直接覆盖掉你刚改好的磁盘文件。
- **应对**：若要用外部脚本修正底层状态，修改完毕后**必须立刻重启**该带有缓存的守护进程。更高级的解法是在服务端内部引入状态交叉校验逻辑（如 Live Task VS Queued Task），让系统自动识别死信并将其淘汰，无需外部人工干预。

## 2. 异步隔离与冷启动防御
- **直觉**：任何潜在耗时、有可能超时阻塞的外部动作（诸如拉取远程 Git 代码、远程服务连通性握手等），**绝对不能**放在系统的同步启动路径或 Socket 激活的第一个请求周期中。
- **应对**：首个初始化必须轻量如飞。沉重的依赖或预热应放入后台线程（`threading.Thread` 等）异步执行。哪怕启动后暂无数据，也比直接阻塞挂死请求并导致调用方 504 Timeout 要强。

## 3. 开机冷启动的避灾重试与限频保护
- **直觉**：系统刚开机拉起的自启定时任务，大概率会遭遇宿主机的底层依赖（DNS解析、代理隧道、外部网卡）尚未完全就绪的情况，进而触发一堆连接瞬时报错。
- **应对**：定时调度器绝不能简单地“触发过一次就无脑标记今日已执行完毕”，这样会造成灾难性的业务漏做（如打卡/抓取失败）。必须精确区分“业务正常终态”和“临时网络异常”。对于后者，应该引入状态机记录（如持久化 JSON 里的 `retry_count`），每隔几分钟轮询重试。为了防止永久断网引发死循环重试，必须设定安全重试上限（如 5 次）。

## 4. 数据分区与前端展示的绝对隔离
- **直觉**：当系统中引入了在逻辑心智上完全不同于已有数据的“新数据类型”或“新分区”（例如，加入长视频，它不同于之前的分季番剧）。
- **应对**：绝不可偷懒直接复用已有的聚合 UI 和分类路由。除了后端增加对应的路由过滤口径（如 `folder=long`），必须将这批旧数据的物理归属地（目录结构/文件位置）也顺势做物理隔离迁移；并为前端大刀阔斧地阉割掉不符合心智的控件（例如隐藏长视频不需要的“下一集”和分类侧边栏）。UI 必须精确适配产品心智，哪怕底层的存储对象是同一个。

## 5. 跨服务聚合：业务只读与状态压缩
- **直觉**：对于暴露在顶层的公共监控大盘（Dashboard），当它需要整合多个下游独立业务系统（如签到系统、爬虫系统）的状态时。
- **应对**：大盘**绝对不应**自行复刻下游业务的爬取代码、抓取 Cookie 或做深度解析，这会导致可怕的代码分裂与凭证泄露。下游业务必须提供专用的“内部聚合只读 API”，大盘只负责做定期轮询与分级缓存。
- **降噪要求**：对于向小白或者非专业运维展示的管理面，永远不要直接透传底层的原始 HTTP 状态码。在前端顶层必须使用状态压缩，把它降维成一两句“可执行的结论”，如：“正常，无需更换配置”。

## 6. 基础设施状态同步与防回归
- **直觉**：任何一个后端服务模块的新增上线或端口下线，仅仅是后台建设的结束，但却只是完整项目闭环的一半。
- **应对**：后端上线后，必须立刻同步去核查前端的“端口总览面板 (Ports Panel)”、“主控制台菜单”等静态可视化入口，保证系统状态在界面层同步反馈。
- **防回归直觉**：在重构公共监控大盘的代码时，必须提前检索架构地图，确认当前大盘承载了哪些既有的监控能力。严防在加新功能的时候，随意或无意地把以前辛苦跑通的模块（如 Aria2 下载状态）给直接剔除了。

## 7. 自动化运维部署脚本的防漂移
- **直觉**：所有的部署、更新、热重载等 Ops 脚本，若包含强依赖某台特定机器、特定用户的绝对路径（如 `/Users/xxx/Documents/...`），它一定会在项目迁移后随时暴雷挂掉，这就是“工作区漂移”。
- **应对**：脚本内部必须使用基于相对位置的环境上下文获取机制。另外，每次写部署脚本要警惕“只传后端不传前端”的历史局限，一定要同时把更新后的静态资源（`server.py` + `index.html` + `style.css` 等）捆绑覆盖，避免部署后前端仍是旧版死灰复燃。
