# 运维网络与代理容器排坑手册 (Domains)

> 涵盖 Mihomo/Clash 配置、代理组行为以及端口映射运维。

## 1. Mihomo/Clash URLTest 组的 Fixed 残留死锁
* **表现**：配置文件里增加了速度更快、延迟更低的高级专线节点订阅。但经过数日运行，发现该 `URLTest` (自动测速选优) 组当前一直死死锁定在一条极其老旧缓慢的节点上，没有自动切到新节点。
* **根因**：如果该代理组在此前曾经被用户通过第三方 UI 面板、或者后台自愈 Bot 脚本，下发过 `PUT /proxies/组名 {"name":"某节点"}` 进行过绝对强行指定，系统就会为该组隐式保留一个 `fixed` 锁定状态。在 `fixed` 存在的情况下，`URLTest` 本身的内置自动重测速切换逻辑会完全失效退避。
* **填坑**：要恢复其弹性选优能力，脚本干预不应使用 `PUT` 继续去抢控制权。而是必须：
  1. 调用 `DELETE /proxies/组名` 请求，清空其固化状态（响应码 204）。
  2. 紧接着调用 `GET /group/组名/delay?...` 主动触发该组内的所有节点立即重测速。
  3. 完成后该代理组即可基于新的测速结果重新自我恢复到 `URLTest` 的低延迟首选逻辑中。

## 2. Ports Panel 与本地监听脱节防呆
* **表现**：成功在本地通过 `docker` 部署起了一个服务（如 Dashboard），`curl 127.0.0.1:8081` 有正常返回。但在导航总览面板 `:8083` 却找不到对应入口，或者之前下线的端口服务在 `:8083` 仍有一个无效点击入口。
* **根因**：许多轻量团队没有接入动态服务注册中心（如 Consul），聚合面板使用的是硬编码或静态配置文件（如 `ports_panel/server.py` 的写死字典）。
* **填坑**：在所有本地后端系统（新增或废弃）上执行操作时，必须强行在脑中串联起静态主面板。服务上线或下线不仅是 `systemctl` 状态的变更，其最终闭环必须包含针对主导航面板相关配置文件中入口条目的新增与剔除。

## 3. 部署脚本的工作区硬编码漂移 (Workspace Drift)
* **表现**：在一台新服务器或者修改过主目录名称（如从 `/Server` 迁往 `/TX`）的环境下，运行原有基于 Python 编写的部署脚本 `_update_xxx.py`，发现线上代码死活没有更新。
* **根因**：老脚本在代码获取阶段硬编码了诸如 `source_dir = '/Users/username/Documents/Server/...'`，导致新目录下的修改代码根本没被打包，而是仍从可能早已陈旧或废弃的老目录下抓取并上传覆盖。
* **填坑**：
  * **脚本重构**：绝不能用硬编码特定用户的绝对路径来组装部署资源，应通过 `os.path.dirname(__file__)` 定位当前环境的根，再使用相对路径组合。
  * **前后端打包齐备**：要核查历史部署脚本，部分项目早期的更新脚本只单独处理后端文件（如 `server.py`），导致新改版的 `index.html` 和 `style.css` 丢失。部署更新必须做到全组件同步覆盖。
