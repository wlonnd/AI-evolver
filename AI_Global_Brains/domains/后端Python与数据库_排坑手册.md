# 后端Python与数据库排坑手册 (Domains)

> 针对 Python 守护进程、Socket 激活、以及多层任务调度和数据库排障的细节记录。

## 1. Systemd Socket 激活机制的冷启动死锁
* **表现**：由 `xxx.socket` 代为监听并拉起 `xxx.service` 的 Python 服务（如复习系统），在服务器长期空闲被唤醒时，首个 HTTP 请求经历了极其夸张的延时（如 2000ms+），导致调用方报 504 Timeout。
* **根因**：服务内部 `server.py` 在初始化主进程的阻塞路径中，放置了如 `os.system("git pull")` 这种重量级网络操作，Socket 接收到连接请求后，必须等待 Python 把上述操作执行完才能响应。
* **填坑**：服务启动主线必须是毫秒级。任何外部依赖拉取，都必须抽离并丢进 `threading.Thread(target=..., daemon=True).start()` 后台异步完成。

## 2. 定时任务(APScheduler等)的开机网络异常与重试
* **表现**：设定在早晨自动触发的签到 Python 任务，如果服务器在那时才刚开机，任务在开机启动后立刻执行，却因 DNS 解析或代理连接失败而报错。且当天余下时间不再触发。
* **根因**：服务刚开机时底层的网络环境尚未完全就绪。代码里只判断“一旦被调度触发，就把今天标记为已执行完毕”。
* **填坑**：必须改写状态判定逻辑。若捕捉到非持久性错误（如 HTTP Timeout / Connection Error），不要写下 `today_done=True` 的状态，而是向持久化文件（`state.json`）中累加 `schedule_retry_count`，并利用 Tick 轮询或延迟器在 5 分钟后发起重试；并在重试达到 5 次上限后熔断报错，防范死循环请求。

## 3. 多层异构组件的 SQL WHERE 条件隔离泄露
* **表现**：一个后台去重系统（包含上游 Scanner 提取任务，和下游 Analyzer 调动 GPU 跑算法）。业务方要求某类长视频目录（`/long/`）免去重以节约算力，并且已经在前端屏蔽了展示。然而系统 GPU 仍被耗尽卡死数小时。
* **根因**：SQL 查询 `WHERE filepath NOT LIKE '%/long/%'` 漏加在了上游 `server_scanner.py` 或下游 `server_analyzer.py` 其中一个地方。导致免除扫描的文件仍然被误选入任务队列进行高耗能提取。
* **填坑**：增加此类特殊忽略规则时，必须排查该表的**所有数据流转口径**。同时，对于已经误入歧途的存量大文件记录，要果断编写修复脚本连接数据库，将其特征字段（如 `phash`）批量置空 `NULL`，强制标定 `checked_for_dups=1`，并通过删除关联副表（`duplicates`）对应行，彻底切断重计算残留，随后必须重启上下游两个 systemd 服务以清除僵尸进程。

## 4. 长生命周期守护进程的 Manifest 冲突
* **表现**：Telegram Bot 长期驻留内存运行，外部管理端页面向 Bot 的 `external_pending.json` 添加了一个链接，却被判定为“陈旧的阻塞/队列中”，并且通过外部脚本去修改 Bot 的清单 JSON 文件删除了阻塞记录后，该记录马上又重新复活。
* **根因**：Python 守护进程通常在启动时将整个 `manifest.json` 读入内存字典缓存，当有新变化时全量写回磁盘。你在磁盘上的外部修改，在下一次落盘时被内存里的旧数据覆盖。
* **填坑**：
  * **方案 A（笨拙安全）**：修改了其依赖的磁盘状态文件后，使用 `systemctl restart tg-userbot` 立即强制重启守护进程重载内存。
  * **方案 B（根治机制）**：在内存守护程序内，加入交叉状态核验函数。在判定该任务属于 `queued` 的同时，去磁盘核查其是否有真实排队中的物理实体存在（如跨库比对 `tg_queue.json`）。如果物理实体已不存在，说明该 `queued` 是死信残留，应由守护程序**自身主动将其清理为 failed 态**，彻底消除对人工干预清理的依赖。
