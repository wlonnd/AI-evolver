# Project Rules (AI 协作最高宪法)

> **严禁违背**：任何接手此项目的 AI，在处理每个请求时，必须严格遵守以下法则。这套规则确保了本项目的“强制工程闭环”。

1. **Start of Task - Mandatory Routing Lookup**: At the start of any new request, you must read `架构地图.md` (or the equivalent root map) section `0. AI 接手必读流程`. If the task involves a concrete object, service, or component, you MUST consult the relevant route-table section before taking action.
2. **Micro-Step Navigation Rule (NO BLIND SEARCHING)**: Before you take ANY sub-step (such as locating a UI file, finding where to edit a parameter, or deploying a change), you MUST FIRST search the main `架构地图.md` or its linked sub-maps for the exact file path and location.
3. **Prohibition of Guesswork**: You are STRICTLY PROHIBITED from using `grep_search`, `find`, or browsing folders blindly to "figure out" where a component is. You must always assume the answer is in the Map documents. Only if you have verified that the documents absolutely do not contain the path, are you allowed to search the codebase.
4. **Auto-Evolution on Micro-Steps**: If you find that a crucial file path or component location is missing from the Maps and you had to search the codebase to find it, you MUST update the corresponding Map document to include this new path/location mapping so future AI won't have to search.
5. **Continuous Context**: Do not rely on your memory. If you forget which file controls a specific component during a long task, go back and read the relevant Map document again.
6. **Mandatory Map Sync Before Task Completion**: For any task that changes architecture, state, or logic, or confirms that the codebase now differs from what the Maps say, you MUST check whether the relevant Map sections need updating and sync them before declaring the task complete. If the map has not been checked and updated where needed, the task is NOT complete.
7. **No Route, No Action**: If an action does not have a clear route in the maps or specialized route-table files, you MUST stop and add or repair the route first. Do not execute ad hoc operational work without a documented route.
8. **Project Registration Before Expansion**: If the task introduces a new long-lived subsystem or route-table domain, you MUST first register it in the main map before treating it as a normal maintenance target.
9. **Mandatory Task Closure Loop**: Every non-trivial task must follow the closure route (e.g., `maps/routes/路由表_AI任务闭环.md`): classify intent, resolve route, check impact, execute, test, update maps/knowledge, then summarize what changed.
10. **Algorithm Rationale Protection**: Before changing any algorithm, threshold, or core rule, you MUST check for rationale and dependencies (e.g., `maps/knowledge/算法约束索引.md`). If the rationale is missing and you discover it, add it before completion.
11. **Experience Lookup and Writeback**: Before debugging or modifying behavior, check `maps/knowledge/经验库.md` for similar incidents. After solving a new or recurring issue, record the symptom, root cause, fix, verification, and related routes.
12. **Map Writing Rules**: Whenever you update any map, route table, or knowledge document, you MUST follow `maps/knowledge/地图写作规则.md` (if it exists) to ensure format and granularity constraints.
13. **Progressive Route Refinement During Work**: While performing a task, if any sub-step exposes that the current route is too coarse, missing a script, or outdated, you MUST refine the corresponding route table before continuing.
14. **Global Knowledge Usage (全局直觉的调用与边界)**: 
    - **Consumer Role (Lazy Loading)**: When encountering strange bugs, read your local `ai_intuition/` folder. **CRITICAL LAZY LOADING RULE**: You MUST ONLY read the specific markdown file(s) that exactly match your current technical domain (e.g., frontend bugs -> read frontend manuals). You are STRICTLY PROHIBITED from blindly listing or reading all files in the `ai_intuition/` folder, to prevent catastrophic context window waste.
    - **Raw Data Provider Role**: You MUST write all concrete, project-specific bugs and fixes to your local `maps/knowledge/经验库.md` (Rule 11).
    - **Prohibition of Distillation**: You are STRICTLY PROHIBITED from extracting or distilling local experience to the global folder yourself. That is the Chief Architect AI's job.
15. **Mandatory Dependency Declaration**: Any introduction of a new external library, API, package (e.g., pip, npm), or runtime environment MUST be registered in the `Runtime & Dependencies` section of the Architecture Map before task completion.
16. **Strict Sandbox Confinement**: All map lookups, modifications, and script executions MUST be strictly confined to the current project's root directory. You are PROHIBITED from reading or modifying root maps of other projects in the workspace unless explicitly instructed.
17. **Directory Granularity Enforcement**: Any newly created subdirectory that represents a distinct functional module, service, or script category (even if nested within an already registered parent like `src/`) MUST be explicitly registered in the `Core Directory Mapping` section of the Architecture Map.
18. **Anti-Sycophancy & Strict Route Rejection (拒绝讨好与无路由任务抗命)**: You are STRICTLY PROHIBITED from accepting or executing ad-hoc, temporary, or 'distraction' tasks (e.g., generating massive random data, writing one-off processing scripts) just to please the user or the system prompt, if those tasks lack a predefined route in the architecture map. You MUST reject them citing Rule 7 (No Route, No Action) to protect your context window and ensure absolute rule compliance (such as Rule 10).
