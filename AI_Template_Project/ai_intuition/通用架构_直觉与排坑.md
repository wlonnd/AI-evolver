# 通用架构 - 直觉与疑难杂症手册

> 本文件仅存放**与具体编程语言无关**的通用原则。例如：数据库设计、并发锁策略、缓存穿透/雪崩防御、防御性编程思想等。

## 1. 防御性编程直觉
*(蒸馏条目追加于此)*
* 【排障直觉】看到UI上的“历史错误”提示，不要立刻假设当前主流程已卡死。先通过底层日志/脚本确认该错误是“历史遗留的僵尸状态”还是“真实的当前阻塞”，再决定是否要重新入队或清理状态。
* 【数据库过滤遗漏】后台扫描器如果增加新路径类型（如`/long/`），原先只排除了`/other/`的 WHERE 查询会引发资源错配甚至打满 GPU。新数据域的引入，必须连带审计所有扫描、分析进程的 SQL 排除条件。

## 2. 并发与一致性直觉
*(蒸馏条目追加于此)*
* 【缓存与状态一致性】当使用 `manifest` 或缓存层做防重逻辑时，如果底层实际任务（如队列文件/存活进程）已经不存在，缓存就会变成“陈旧阻塞”。修复时不能只改磁盘，必须确保进程内缓存（内存状态）与磁盘强一致；最佳实践是读取时交叉校验缓存和底层真实状态。
* 【性能与状态同步】服务启动路径中绝对避免同步的长耗时操作（如网络请求、Git Pull），必须解耦到后台线程。多设备多端网页如果存在本地缓存，必须通过页面可见性或定时轮询主动回拉服务端最新状态，避免本地状态陈旧。

## 3. 容错与重试机制
*(蒸馏条目追加于此)*
* 【后端/定时任务鲁棒性】开机启动时（Boot）网络、DNS极有可能未就绪。定时调度不仅要看“今天是否跑过”，还要对“网络临时失败”做延时重试，但同时必须有 **限次/限频** 机制（如最大重试5次），否则极易造成请求风暴死循环。临时失败只递增重试次数不写完成日期；成功或超出限次后，方可写入完成日期。

## 4. 系统演进与耦合设计
* 【架构解耦与看板】当 Dashboard 需要展示其他业务域的数据时，绝不能让 Dashboard 重写抓取逻辑或直接持有时效 Cookie。正确做法是让业务域服务自己暴露只读 `/snapshot` 接口，Dashboard 仅去拉取并做适度分级缓存。
* 【系统文档化演进】任何自演进系统/AI闭环操作，必须要有明确的入口和清晰的路由表，且路由表在执行长任务期间发现颗粒度太粗时，必须“立即细化”而不是等事后补。写地图也需要元规则，才能形成闭环。
* 【路由与数据分区】在现有资源分类外新增分类（如“长视频”）时，必须从入口API、前端页面、落盘目录进行全链路三层分离。操作具体数据项迁移时，先用 dry-run 确认实际源路径，再迁移并更新DB，防止盲目假设其已被分拣。
